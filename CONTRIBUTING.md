# Contributing to Antinode Norma

Thank you for your interest in contributing to **Antinode Norma**! We welcome all kinds of contributions – from fixing typos to adding new features.

---

## Getting Started

- Read the [README.md](../README.md) to understand the project overview and quick start.
- Check the [Client Usage Guide](CLIENT_USAGE.md) for detailed setup instructions.
- Review the [Testing Guide](TESTING.md) to learn how to run and extend the test suite.

If you're stuck on a technical issue, please open a [GitHub Issue](https://github.com/antinodelabs/antinode-norma/issues).

---

## Ways to Contribute

- **Fix a bug or typo** – Open a pull request directly.
- **Suggest a new feature or improvement** – Open an issue first to discuss the change.
- **Add a new LLM provider** – Extend `llm_factory.py` to support additional providers.
- **Improve the INVEST quality checks** – Enhance `quality.py` with better heuristics or new criteria.
- **Add a new connector** – Contribute connectors for Azure Boards, GitHub Issues, Trello, etc., by implementing the connector pattern.
- **Improve documentation** – Clarify usage, add examples, or translate guides.
- **Report outdated information** – Open an issue.- Use the issue templates in `.github/ISSUE_TEMPLATE/` for bugs and feature requests.
---

## Development Workflow

1. **Fork** the repository.
2. **Create a feature branch** – `git checkout -b feature/your-feature-name`
3. **Make your changes** – Follow the code style and add tests.
4. **Run tests** – Ensure all tests pass: `pytest`
5. **Commit** – Write a clear commit message.
6. **Push** – `git push origin feature/your-feature-name`
7. **Open a Pull Request** – Target the `develop` branch (or `main` for hotfixes).

---

## Pull Request Guidelines

- **Keep changes focused** – One logical change per PR.
- **Update documentation** – If adding or modifying functionality, update relevant docs (README, CLIENT_USAGE, TESTING).
- **Add tests** – New features should include unit and/or integration tests.
- **Update the "Last Updated" date** in any documentation file you modify.
- **Request review** – From at least one maintainer.

---

## Code Style & Quality

- **Python** – Follow [PEP 8](https://peps.python.org/pep-0008/).
- **Formatting** – Use `ruff` for consistent formatting:
  ```bash
  python -m ruff format antinode_norma/ tests/
  ```
- **Linting** – Run `ruff` and `mypy` to catch issues.
- **Type hints** – Use type annotations for all function signatures.
- **Docstrings** – Write clear docstrings for public functions and classes.

---

## Tone & Style (for Documentation)

- Write for clarity, not completeness. Shorter is better.
- Use active voice. *"The agent generates a feature file"* not *"A feature file is generated by the agent."*
- Be empathetic – this is guidance, not a rulebook.

---

## Review Process

Maintainers review PRs weekly. If urgent, mention `@antinodelabs/eng` in the PR or reach out via email.
